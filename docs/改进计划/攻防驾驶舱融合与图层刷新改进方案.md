# 攻防驾驶舱融合与图层刷新改进方案（Mock数据版）

## 1. 问题与现状

### 1.1 你提出的问题
1. 攻击路径推演与防御策略仿真目前是独立页面（`/cockpit/attack-path`、`/cockpit/defense`），未在同一网络沙盘中联动展示。
2. 驾驶舱图层过滤（含模型/模式切换）后画布刷新不稳定，出现节点缺失。

### 1.2 代码现状（关键文件）
- 路由将三块拆开：`demo/router/index.tsx`
  - `/cockpit` -> `CockpitPage`
  - `/cockpit/attack-path` -> `AttackPathSimulation`
  - `/cockpit/defense` -> `DefenseStrategy`
- 驾驶舱主页仅含图层控制+沙盘：`demo/modules/cockpit/CockpitPage.tsx`
- 攻击推演单独页面：`demo/modules/cockpit/AttackPathSimulation.tsx`
- 防御仿真单独页面：`demo/modules/cockpit/DefenseStrategy.tsx`
- 图层渲染逻辑：`demo/modules/cockpit/components/SandboxCanvas.tsx`
- 图层控制器：`demo/modules/cockpit/components/LayerController.tsx`

## 2. 改造目标

### 2.1 业务目标
- 统一入口：所有攻防推演都在 `/cockpit` 同一沙盘中完成。
- 统一视图：攻击路径、防御部署、攻防结果（阻断/绕过/损伤）在同一画布叠加显示。
- 统一数据流：基于 Mock 数据形成“攻击推演 -> 防御干预 -> 态势结果”的闭环。

### 2.2 技术目标
- 图层/模式切换后画布必刷新、节点不丢失。
- 推演状态可重放、可切换方案、可比较结果。
- 保持现有模块可复用，降低重写量。

## 3. 改进方案

### 3.1 方案总览：单页驾驶舱编排（推荐）
在 `CockpitPage` 内新增“推演模式 Tab（或 Segment）”，不再跳独立页面：
- `沙盘总览`
- `攻击推演`
- `防御仿真`
- `攻防联动结果`

核心原则：
- 右侧始终是同一个 `SandboxCanvas` 实例（同一张图）。
- 左侧/下方面板按模式切换，输出统一写入“驾驶舱编排状态”。

### 3.2 统一状态编排（新增 store）
新增 `demo/store/useCockpitOrchestratorStore.ts`（建议）：
- 基础上下文：`sceneId`、`activeMode`、`layerConfig`、`modelProfile`。
- 攻击态：`selectedAttackPath`、`attackTimeline`、`compromisedNodes`。
- 防御态：`selectedDefenseScenario`、`deployments`、`defenseEvents`。
- 联动结果：`blockedPhases`、`bypassedPhases`、`finalDamage`、`compareMetrics`。

数据来源全部使用现有 Mock：
- 场景：`mock/dynamic/scenes.ts`
- 攻击路径：`mock/dynamic/attackPaths.ts`
- 防御场景：`mock/dynamic/defenseScenarios.ts`

### 3.3 页面结构调整
1. `CockpitPage` 升级为“编排页”，承接原 `AttackPathSimulation` 和 `DefenseStrategy` 的配置能力。
2. 将 `AttackPathSimulation` / `DefenseStrategy` 中“配置与结果组件”拆为可嵌入子面板，避免重复逻辑。
3. 路由兼容：
   - `/cockpit/attack-path`、`/cockpit/defense` 保留但重定向到 `/cockpit?mode=attack|defense`。

### 3.4 图层刷新问题修复（重点）

#### 发现的高风险点
1. `SandboxCanvas` 初始化 `useEffect` 依赖 `onNodeClick/onNodeContextMenu`，父组件每次重渲染会触发画布销毁重建，容易导致节点闪断或缺失。
2. `CockpitPage` 的 `setLayerConfig({...layerConfig})` 为非函数式更新，快速切换时可能丢状态。
3. 图层切换主要靠 `opacity/show/hide`，未形成“过滤后数据 -> 统一刷新”的稳定链路。

#### 修复策略
1. **稳定 Graph 生命周期**
   - `SandboxCanvas` 初始化 effect 仅依赖容器与尺寸，不再依赖点击回调。
   - 回调通过 `ref` 传递（当前已有雏形），避免反复重建 graph。
2. **函数式状态更新**
   - `setLayerConfig(prev => ({ ...prev, [layer]: enabled }))`
   - `setLayerConfig(prev => ({ ...prev, [layer]: true }))`
3. **数据驱动刷新**
   - 在 `CockpitPage` 里用 `useMemo` 产出 `visibleNodes/visibleEdges`（按主图层 + 叠加层 + 模式过滤）。
   - `SandboxCanvas` 统一使用 `graph.changeData({ nodes, edges })`（或 `data+render`）并立即 `paint/refresh`。
4. **模式与图层解耦**
   - “推演模式”决定业务语义（攻击/防御/联动）；
   - “图层过滤”只决定可视化维度（业务/网络/攻防）。
   - 两者组合后统一产出可视数据集。

### 3.5 Mock联动逻辑（实现口径）
- 攻击推演执行后，生成：
  - `compromisedNodes`
  - `attackPathEdges`
  - `timelineEvents`
- 选定防御方案后，对每个 phase 应用 `effectivenessScore`：
  - `blocked | detected | bypassed`
- 联动结果回写沙盘：
  - 节点状态（在线/降级/失陷）
  - 边样式（攻击链高亮、被阻断标记）
  - 统计面板（成功率降低、损伤变化）

- 新建编排 store 与类型。
- 路由兼容重定向到 `/cockpit`。
- 输出基础 `mode` 参数解析（URL query）。

- `CockpitPage` 增加模式 Tab 和配置面板容器。

- 建立 attack + defense 的联动计算函数。
- 生成联动结果与时间轴，驱动沙盘统一渲染。

### P3（0.5天）：图层刷新修复
- 稳定 `SandboxCanvas` 生命周期。
- 改函数式状态更新，新增数据驱动刷新链路。
- 修复“切换后节点缺失”。

### P4（0.5天）：回归与验收
- Playwright 回归（新增/更新）：
  - 切换 `business/network/attack` 图层节点数不为0；
  - 切换模式后画布节点总数与场景一致；
  - 攻击执行后高亮路径出现；
  - 防御执行后阻断/绕过标记变化。
- 产出测试记录与截图。

## 5. 验收标准
1. `/cockpit` 单页可完成攻击推演、防御仿真、联动结果查看。
2. `/cockpit/attack-path`、`/cockpit/defense` 不再出现独立业务页面（重定向至 cockpit 模式）。
3. 连续切换图层/模式 20 次以上，画布无节点缺失、无空白状态。
4. Mock 数据联动后，指标与画布状态一致（阻断率、受损节点数等）。

## 6. 风险与应对
- 风险：旧页面逻辑耦合较深，直接搬迁可能引入回归。
  - 应对：先做“面板复用+状态编排”，不一次性重写所有组件。
- 风险：G6 渲染与 React 状态节奏不同步。
  - 应对：统一通过“可视数据集版本号 + changeData”刷新，禁止隐式重建。
- 风险：Mock 数据字段命名不一致（如 nodeId/sourceNodeId/sourceId）。
  - 应对：先做 adapter 层，避免在组件内到处做兼容判断。

## 7. 建议的交付顺序
1. 先交付 `/cockpit` 融合版可用流（攻击+防御+联动+图层稳定）。
2. 再清理旧独立页，仅保留兼容重定向。
3. 最后补全自动化回归用例，锁定后续改动风险。
